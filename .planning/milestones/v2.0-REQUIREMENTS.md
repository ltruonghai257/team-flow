## Milestone v2.1: Open WebUI-Style Project Structure Refactor

### Goal

Refactor TeamFlow into a clearer Open WebUI-inspired FastAPI/SvelteKit structure without changing product behavior.

### Structure

- [ ] **STRUCT-01**: Developer can understand the repo layout from a documented target structure that maps current TeamFlow folders to Open WebUI-inspired folders.
- [ ] **STRUCT-02**: Existing behavior boundaries are explicitly protected before code moves begin: API routes, auth/session behavior, Svelte routes, WebSocket chat, scheduler jobs, AI task input, Docker runtime, and Alembic migrations.
- [ ] **STRUCT-03**: Refactor work is split into small migration slices so imports can be updated and verified incrementally.

### Backend

- [ ] **BACK-01**: Backend package layout follows the Open WebUI pattern: package root, `routers`, domain/model modules, schemas, config, migrations, utils, socket/websocket, and app entrypoint.
- [ ] **BACK-02**: Large backend modules are separated by responsibility where useful, especially `models.py` and `schemas.py`, without adding speculative abstractions.
- [ ] **BACK-03**: FastAPI router registration, dependency imports, scheduler startup/shutdown, rate limiting, CORS config, and health endpoint continue to behave as before.
- [ ] **BACK-04**: Alembic config and migration imports still run against the new package paths without rewriting migration history.
- [ ] **BACK-05**: Backend tests import the new package paths and pass after the move.

### Frontend

- [ ] **FRONT-01**: Frontend `src/lib` follows the Open WebUI-style grouping: `apis`, `components`, `stores`, `types`, and `utils`.
- [ ] **FRONT-02**: API client code is split into feature-focused modules under `src/lib/apis` while preserving existing request behavior and auth handling.
- [ ] **FRONT-03**: Shared frontend types are centralized under `src/lib/types` and reused by routes/components where it reduces duplication.
- [ ] **FRONT-04**: Svelte routes keep their current URLs and user-facing behavior after import/path changes.
- [ ] **FRONT-05**: Existing component groups remain feature-oriented and no UI redesign is introduced during the structural refactor.

### Runtime

- [ ] **RUN-01**: Dockerfile, compose/dev scripts, Azure runtime entrypoints, and uvicorn target point to the refactored backend app.
- [ ] **RUN-02**: Frontend build/check commands continue to work with the refactored file layout.
- [ ] **RUN-03**: No new Python or frontend dependencies are added unless a moved import cannot be resolved without one.

### Verification

- [ ] **VERIFY-01**: Backend test suite passes after backend restructuring.
- [ ] **VERIFY-02**: Frontend `check` and production build pass after frontend restructuring.
- [ ] **VERIFY-03**: Targeted smoke checks cover login/session, task board load, AI task input, WebSocket chat connection, scheduler/notifications, and health endpoint.
- [ ] **VERIFY-04**: A migration guide or developer note documents old-to-new import paths and any temporary compatibility shims.

- Resume paused v2.0 feature work: team hierarchy, custom statuses, advanced KPI dashboard, and reminders.
- Convert the repo root to match Open WebUI exactly by moving `frontend/src` to root `src`; this is deferred unless the Docker/build impact is explicitly accepted.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Product feature changes | This milestone is a structural refactor only. |
| Database schema redesign | Existing data model behavior must remain stable. |
| UI redesign | Import and folder moves should not alter visual behavior. |
| Exact Open WebUI clone | TeamFlow should borrow structure patterns, not adopt unrelated AI-platform architecture. |
| New dependency stack | Refactor should use existing FastAPI/SvelteKit/PostgreSQL/Tailwind stack. |
