<task id="18-02-01" type="execute">
  <name>Add transition enforcement helpers</name>
  <read_first>backend/app/routers/tasks.py, backend/app/routers/statuses.py, backend/app/schemas.py</read_first>
  <action>
In `backend/app/routers/tasks.py`, add focused helpers near `_resolve_custom_status`:

- Resolve the task's current custom status if `task.custom_status_id` is set.
- If missing, use existing legacy-status mapping for the task's effective project/sub-team before enforcement.
- Resolve the target custom status from explicit `custom_status_id` or legacy `status`.
- Determine the effective target status set using the target project when `project_id` is changed in the same update.
- Load transitions for that status set.
- If no transition rows exist, allow free movement.
- If current and target custom status IDs are equal, allow as a no-op.
- If rows exist, require an exact `(from_status_id, to_status_id)` edge.

Keep helpers small and unit-testable through endpoint tests. Do not block unrelated task field edits when status/project is unchanged.
  </action>
  <acceptance_criteria>
    - `backend/app/routers/tasks.py` contains a transition enforcement helper
    - Helper treats empty transition graph as unrestricted
    - Helper allows no-op status updates
    - Helper only runs for status changes or project changes that change status context
  </acceptance_criteria>
</task>

Extend project override lifecycle in `backend/app/routers/statuses.py`:

- When creating a project override, copy statuses first.
- Build a source-to-target status ID map by slug from effective default statuses to project override statuses.
- Copy transition rows where both endpoints map to copied statuses.

Keep project override behavior as an independent snapshot after creation.
  </action>
  <acceptance_criteria>
    - Project override creation copies matching transition rows
    - Copied transition rows point at project override status IDs, not default status IDs
    - Reverting project override removes project-specific transition rows
    - Existing project overrides do not receive later default transition changes
  </acceptance_criteria>
</task>

<task id="18-02-04" type="execute">
  <name>Clean up transitions on delete and hide archived transitions</name>
  <read_first>backend/app/routers/statuses.py, backend/app/models.py, backend/tests/test_status_sets.py</read_first>
  <action>
Update status delete/archive behavior:

- Hard delete removes transitions where the deleted status is either `from_status_id` or `to_status_id`.
- `move_delete` also removes transitions involving the deleted status after task reassignment.
- Archive keeps transition rows in the DB but hides them from default transition API responses.
- If a status is unarchived later, existing transition rows become visible again when both endpoints are active.

Use explicit cleanup queries if cascade configuration does not cover both endpoint relationships.
  </action>
  <acceptance_criteria>
    - Hard delete leaves no active transition rows involving deleted status IDs
    - Archive does not delete transition rows
    - Default transition GET hides archived-linked rows
    - `include_archived=true` can return archived-linked rows
  </acceptance_criteria>
</task>
