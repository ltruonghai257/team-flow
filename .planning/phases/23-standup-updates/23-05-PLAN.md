# Plan 23-05: Standup Template Field Types

## Context

Gap closure for Phase 23. The current implementation uses simple text fields for all template fields. This plan adds support for different field types (text, datetime, rich text/markdown) to make standup posts more flexible and structured.

## Goal

Enable standup template fields to have different types (text, datetime, rich text/markdown) with appropriate input and display handling in both backend and frontend.

## Requirements

**FT-01**: Template fields can have types: `text`, `datetime`, `richtext`
**FT-02**: Backend stores field type mapping in `standup_templates.field_types` (JSONB)
**FT-03**: Backend validates field values against their types in POST/PATCH
**FT-04**: Frontend StandupForm renders appropriate input per field type (textarea, datetime-local, markdown editor)
**FT-05**: Frontend StandupCard displays datetime fields in readable format (YYYY-MM-DD HH:MM)
**FT-06**: Frontend StandupCard renders richtext fields safely with marked + dompurify
**FT-07**: Frontend template editor allows selecting field type for each field
**FT-08**: Existing text fields are migrated to type `text` by default

## Success Criteria

1. A supervisor can configure a template with mixed field types (e.g., "Blockers" as text, "Meeting Time" as datetime, "Notes" as richtext)
2. The StandupForm renders datetime pickers for datetime fields and markdown editors for richtext fields
3. Submitted posts store values in the correct format (ISO datetime for datetime, raw markdown for richtext)
4. The StandupCard displays datetime fields in human-readable format and renders richtext safely with marked/dompurify
5. Existing posts continue to work (default type `text`)

## Implementation Notes

### Backend

**StandupTemplate model**:
- Add `field_types` column (JSONB): `{"field_name": "text|datetime|richtext", ...}`
- Default: empty dict `{}` (all fields default to `text`)

**StandupPost model**:
- `field_values` remains JSONB (stores raw values)
- No schema change needed - values are validated at API layer

**Schemas**:
- `TemplateOut`: Add `field_types: Dict[str, str]`
- `TemplateUpdate`: Add `field_types: Optional[Dict[str, str]]` with validation that values are valid types
- `StandupPostCreate`: Validate `field_values` against template field types (datetime fields must be valid ISO 8601)

**Router**:
- `GET /template`: Return `field_types` in response
- `PUT /template`: Accept `field_types` in payload, validate type enum
- `POST /`: Validate datetime fields are valid ISO 8601 before storing

**Migration**:
- Add `field_types` column to `standup_templates` (JSONB, default `{}`)
- No data migration needed (empty dict means all fields default to `text`)

### Frontend

**StandupForm**:
- Accept `fieldTypes: Record<string, string>` prop (from template)
- For each field, render input based on type:
  - `text`: `<textarea>` (existing)
  - `datetime`: `<input type="datetime-local">`
  - `richtext`: `<textarea>` with markdown preview (use marked + dompurify for preview)
- For datetime fields, convert to ISO format before sending to API

**StandupCard**:
- Accept `fieldTypes: Record<string, string>` prop (from template)
- For each field value in read mode:
  - `text`: whitespace-pre-wrap (existing)
  - `datetime`: format using date-fns `format(parseISO(value), 'PPp')` (e.g., "Apr 28, 2026, 2:15 PM")
  - `richtext`: render with marked + dompurify (XSS-safe)

**Template editor (+page.svelte)**:
- Add type dropdown for each field in template editor
- Options: Text, Date/Time, Rich Text
- Send `field_types` in PUT /template payload

**Store**:
- Add `fieldTypes: Record<string, string>` to `updatesStore` state
- Populate from GET /template response
