<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| command-view API to browser | Aggregated milestone, task, and decision data crosses from backend scope rules into a high-signal UI response |
| milestone/task/decision joins | Joined data can accidentally leak out-of-scope records if filters are incomplete |
| derived planning-state helpers | Incorrect derivation could misclassify work and expose misleading operational state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-04 | I | `GET /api/milestones/command-view` | mitigate | Reuse existing authenticated filters and assert in tests that only visible milestones, tasks, and decisions are included in the aggregated response |
| T-28-05 | T | derived planning-state helper logic | mitigate | Encode the exact `committed` and `active` rules in one helper and cover them with fixture-based tests for all four lanes |
| T-28-06 | D | command-view aggregation query | mitigate | Use eager loading and one dedicated derived response instead of per-milestone N+1 queries so the page does not degrade under normal milestone counts |
</threat_model>
