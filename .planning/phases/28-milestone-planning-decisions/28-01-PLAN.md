| Boundary | Description |

| client to milestone decision API | Untrusted browser input crosses into decision CRUD routes |
| decision to linked task relationship | Optional `task_id` can try to reference a task outside the milestone/project scope |
| migration to existing work tables | New decision table joins against live milestone/task records and must preserve existing data integrity |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-01 | E | `backend/app/routers/milestones.py` decision CRUD routes | mitigate | Reuse authenticated milestone access and reject writes to missing or out-of-scope milestones before mutating decisions |
| T-28-02 | T | `MilestoneDecision.task_id` linkage | mitigate | Validate linked tasks belong to the same milestone or same project context before create/update, and cover rejection with backend tests |
| T-28-03 | D | Alembic migration and ORM relationships | mitigate | Keep the new table narrowly scoped, indexed by `milestone_id`, and avoid cascade rules that could delete unrelated tasks or milestones unexpectedly |
</threat_model>
