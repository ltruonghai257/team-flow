# Plan 15-01 — Backend Status Schema, Migration, and Seed Backfill

<objective>
Create the DB-backed status foundation: `StatusSet`, `CustomStatus`, `Task.custom_status_id`, default status seed data, and task backfill while preserving legacy `Task.status` and existing `completed_at` values.
</objective>

<context>
This plan covers STATUS-03 and the schema foundation for STATUS-04. It must land before any API or UI work.
</context>

<must_haves>
- D-01: default status sets are scoped per sub-team.
- D-04: projects without custom statuses inherit sub-team defaults.
- D-05: statuses have stable slugs.
- D-07: multiple statuses may have `is_done = true`.
- D-08: completion transitions will be based on `is_done`; schema must expose it.
- D-17: legacy `Task.status` remains during dual-write.
- STATUS-03: existing enum statuses are created as DB records and tasks are backfilled with no data loss.
- STATUS-04: `CustomStatus.is_done` exists for completion logic and Phase 16 KPI joins.
</must_haves>

<task id="15-01-01" type="execute">
  <name>Add status set models and task FK</name>
  <read_first>backend/app/models.py, .planning/phases/15-custom-kanban-statuses/15-RESEARCH.md, .planning/phases/15-custom-kanban-statuses/15-CONTEXT.md</read_first>
  <action>
In `backend/app/models.py`, add:

- `class StatusSetScope(str, enum.Enum)` with values `sub_team_default = "sub_team_default"` and `project = "project"`.
- `class StatusSet(Base)` with `__tablename__ = "status_sets"` and columns:
  - `id = Column(Integer, primary_key=True, index=True)`
  - `scope = Column(Enum(StatusSetScope), nullable=False, index=True)`
  - `sub_team_id = Column(Integer, ForeignKey("sub_teams.id"), nullable=True, index=True)`
  - `project_id = Column(Integer, ForeignKey("projects.id"), nullable=True, index=True)`
  - `created_at` and `updated_at` using the same naive UTC lambda pattern as `Task`.
- `class CustomStatus(Base)` with `__tablename__ = "custom_statuses"` and columns:
  - `id = Column(Integer, primary_key=True, index=True)`
  - `status_set_id = Column(Integer, ForeignKey("status_sets.id"), nullable=False, index=True)`
  - `name = Column(String, nullable=False)`
  - `slug = Column(String, nullable=False, index=True)`
  - `color = Column(String, nullable=False, default="#64748b")`
  - `position = Column(Integer, nullable=False, default=0)`
  - `is_done = Column(Boolean, nullable=False, default=False, index=True)`
  - `is_archived = Column(Boolean, nullable=False, default=False, index=True)`
  - `legacy_status = Column(Enum(TaskStatus), nullable=True, index=True)`
  - `created_at` and `updated_at` using the same pattern as `Task`.
- `Task.custom_status_id = Column(Integer, ForeignKey("custom_statuses.id"), nullable=True, index=True)`.
- Relationships:
  - `StatusSet.statuses = relationship("CustomStatus", back_populates="status_set", order_by="CustomStatus.position", cascade="all, delete-orphan")`
  - `StatusSet.sub_team = relationship("SubTeam")`
  - `StatusSet.project = relationship("Project")`
  - `CustomStatus.status_set = relationship("StatusSet", back_populates="statuses")`
  - `CustomStatus.tasks = relationship("Task", back_populates="custom_status")`
  - `Task.custom_status = relationship("CustomStatus", back_populates="tasks")`
  </action>

    - `backend/app/models.py` contains `class CustomStatus(Base)`
    - `backend/app/models.py` contains `custom_status_id = Column(Integer, ForeignKey("custom_statuses.id")`
    - `backend/app/models.py` still contains `status = Column(Enum(TaskStatus)`
    - `backend/app/models.py` contains `is_done = Column(Boolean`

<task id="15-01-02" type="execute">
  <name>Create explicit Alembic migration with seed and backfill</name>
  <read_first>backend/alembic/versions/7b9f1c2d3e4a_add_task_type.py, backend/alembic/versions/6ff5de88b5d6_add_sprint_model.py, backend/app/models.py</read_first>
  <action>
Create `backend/alembic/versions/*_add_custom_statuses.py`.

The migration must:

1. Create `status_sets` with columns matching the model.
2. Create `custom_statuses` with columns matching the model.
3. Add nullable `tasks.custom_status_id`.
4. Create indexes for:
   - `status_sets.scope`
   - `status_sets.sub_team_id`
   - `status_sets.project_id`
   - `custom_statuses.status_set_id`
   - `custom_statuses.slug`
   - `custom_statuses.is_done`
   - `custom_statuses.is_archived`
   - `tasks.custom_status_id`
5. Seed one `sub_team_default` set for each existing row in `sub_teams`.
6. Seed one fallback `sub_team_default` set with `sub_team_id = NULL` for unscoped projects/tasks.
7. For every seeded set, insert these statuses exactly:
   - `To Do`, slug `todo`, color `#64748b`, position `0`, `is_done = false`, `legacy_status = todo`
   - `In Progress`, slug `in_progress`, color `#0ea5e9`, position `1`, `is_done = false`, `legacy_status = in_progress`
   - `Review`, slug `review`, color `#f59e0b`, position `2`, `is_done = false`, `legacy_status = review`
   - `Done`, slug `done`, color `#10b981`, position `3`, `is_done = true`, `legacy_status = done`
   - `Blocked`, slug `blocked`, color `#f43f5e`, position `4`, `is_done = false`, `legacy_status = blocked`
8. Backfill `tasks.custom_status_id` by matching task legacy `status` to the status set for the task's project sub-team; use fallback set for tasks without a project or project sub-team.
9. Preserve all existing `tasks.completed_at` values.

Do not drop, rename, or alter `tasks.status`.
  </action>
  <acceptance_criteria>
    - A file under `backend/alembic/versions/` contains `create_table("status_sets"`
    - The migration contains `create_table("custom_statuses"`
    - The migration contains `add_column("tasks", sa.Column("custom_status_id"`
    - The migration contains each slug string: `"todo"`, `"in_progress"`, `"review"`, `"done"`, `"blocked"`
    - The migration contains `is_done`
    - The migration does not contain `drop_column("tasks", "status")`
  </acceptance_criteria>
</task>
