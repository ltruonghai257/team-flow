Run the Playwright suite. Playwright config uses `webServer: bun run dev` on port 5173 with `reuseExistingServer: true`, so either approach works:

A. Cold run (Playwright starts the dev server itself):
```
cd frontend && bunx playwright test --reporter=list,html
```

B. Warm run (you have already started `bun run dev` and the backend on :8000):
```
cd frontend && bunx playwright test --reporter=list,html
```

Fallbacks in priority order if `bun`/`bunx` unavailable:

1. `cd frontend && npx playwright test --reporter=list,html`
2. CI-recorded Playwright run (paste the summary line; reference the artifact URL).

<task id="22-03-02" type="execute">
  <name>Manual smoke: login, session, task board, AI input</name>
  <files>.planning/phases/22-runtime-integration-regression-verification/22-VERIFICATION.md</files>
  <read_first>.planning/phases/22-runtime-integration-regression-verification/22-VERIFICATION.md, backend/app/routers/auth.py, backend/app/routers/ai.py, .planning/phases/19-refactor-map-safety-baseline/19-SAFETY-BASELINE.md</read_first>
  <action>
Exercise four flows against a running stack and record each as its own row under `## Manual Smoke (Plan 22-03)` in `22-VERIFICATION.md`. Do not paste tokens, cookies, or AI provider keys; record status codes, latency, and a redacted snippet.

For each flow, use the canonical command first; if blocked, document blocker + fallback (D-11).

1. **Login (`POST /api/auth/token`)** — canonical:
   ```
   curl -i -X POST http://localhost:8000/api/auth/token \
     -H "Content-Type: application/x-www-form-urlencoded" \
     -d "username=<seeded-user>&password=<seeded-pass>"
   ```
   Expected: HTTP 200, `Set-Cookie: access_token=...; HttpOnly; SameSite=lax`. Record status + cookie name (not value).

2. **Session (`GET /api/auth/me`)** — canonical:
   ```
   curl -i http://localhost:8000/api/auth/me -b "access_token=<token>"
   ```
   Expected: HTTP 200, JSON with `email` field. Record status + a single non-PII field name list.

3. **Task board load (`/tasks` route)** — canonical: open `http://localhost:5173/tasks` in a browser with the auth cookie present. Confirm columns render and at least one task card appears (or empty-state if seed has none). Record `passed` + a one-line note (e.g., "3 columns, 7 tasks").

4. **AI task input** — canonical:
   ```
   curl -i -X POST http://localhost:8000/api/ai/breakdown \
     -H "Content-Type: application/json" \
     -b "access_token=<token>" \
     -d '{"prompt": "Plan a 1-week sprint to fix the login flow"}'
   ```
   (Verify the exact endpoint path against `backend/app/routers/ai.py` before running.) Expected: HTTP 200 and a JSON list of sub-tasks. If the AI provider key is unset, record HTTP 500/422 and document the fallback (mocked client log line). Do not record the provider response body verbatim — record only that a non-empty list of sub-task strings was returned and the count.

Record format:

```
## Manual Smoke (Plan 22-03)

| Flow | Requirement | Canonical Path | Status | Snippet | Blocker / Fallback | Timestamp |
|------|-------------|----------------|--------|---------|--------------------|-----------|
| Login | VERIFY-03 | POST /api/auth/token | 200 | Set-Cookie: access_token=…; HttpOnly | — | <ISO> |
| Session | VERIFY-03 | GET /api/auth/me | 200 | fields: id,email,full_name,is_admin | — | <ISO> |
| Task board | VERIFY-03 | GET /tasks (route) | passed | 3 columns, 7 tasks | — | <ISO> |
| AI input | VERIFY-03 | POST /api/ai/breakdown | 200 | sub_tasks length=5 | — | <ISO> |
```
