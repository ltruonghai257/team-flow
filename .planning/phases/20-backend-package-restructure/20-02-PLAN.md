# Phase 20 Plan 02: Domain Model Package Split

<objective>
Convert the large backend model module into a domain model package while keeping `app.models` as an aggregate compatibility surface for Alembic, tests, and existing imports.
</objective>

<task id="20-02-01" type="execute">
  <name>Inventory current models and choose domain clusters</name>
  <files>backend/app/models/</files>
  <read_first>backend/app/models.py, backend/app/schemas.py, backend/app/routers, backend/alembic/env.py</read_first>
  <action>
Create the model split map before moving definitions. Use these default clusters unless the executed Phase 19 backend map gives a better compatible breakdown:

- `users.py`: user and invite/account-adjacent models and enums
- `teams.py`: sub-team and reminder-setting/proposal models
- `work.py`: projects, milestones, sprints, tasks, dependencies, status sets, custom statuses, transitions, and work-related enums
- `notifications.py`: schedules, event notifications, notification enums
- `communication.py`: chat/channel models
- `ai.py`: AI conversation/message models

Record any deviations in a short comment in `backend/app/models/__init__.py` or in the Phase 20 migration guide later. Keep tightly coupled work-management models together to avoid relationship cycles.
  </action>
  <acceptance_criteria>
    - Model cluster choice is explicit before moving definitions
    - Work-management models remain grouped together unless Phase 19 map says otherwise
    - No model behavior or table name changes are introduced
  </acceptance_criteria>
  <verify>`rtk proxy rg "class .*\\(Base\\)" backend/app/models.py` before moving, then compare the class list after the package split</verify>
  <done>Model split map is clear enough to move definitions without losing a model.</done>
</task>

<task id="20-02-02" type="execute">
  <name>Replace `models.py` with `models/` package and aggregate exports</name>
  <files>backend/app/models.py, backend/app/models/__init__.py, backend/app/models/*.py</files>
  <read_first>backend/app/models.py, backend/app/db/database.py</read_first>
  <action>
Move model definitions from `backend/app/models.py` into `backend/app/models/` modules.

Important filesystem note: avoid leaving both `backend/app/models.py` and `backend/app/models/` as competing import candidates. The final import surface must be the package `backend/app/models/__init__.py`, so `from app.models import User` continues to work.

Implementation requirements:

- Keep table names, column names, constraints, relationships, enum values, and defaults unchanged.
- Import `Base` from the canonical database module, e.g. `app.db.database`.
- Use string-based SQLAlchemy relationships where helpful to avoid import cycles.
- Add `backend/app/models/__init__.py` aggregate exports for every model/enum previously imported from `app.models`.
- Preserve `__all__` or equivalent explicit exports if useful for compatibility.
  </action>
  <acceptance_criteria>
    - `from app.models import User, Task, Project, EventNotification` works
    - Every previous SQLAlchemy model class still imports from `app.models`
    - No table name or enum value changes are introduced
    - Work-management model relationships import without circular errors
  </acceptance_criteria>
  <verify>`cd backend && rtk proxy python -c "import app.models as m; required = ['User','SubTeam','Project','Milestone','Sprint','Task','StatusSet','CustomStatus','StatusTransition','EventNotification','Schedule','AIConversation']; missing = [name for name in required if not hasattr(m, name)]; assert not missing, missing"`</verify>
  <done>`app.models` is a package facade backed by domain model modules.</done>
</task>

<task id="20-02-03" type="execute">

  <files>backend/alembic/env.py</files>
  <read_first>backend/alembic/env.py, backend/app/models/__init__.py, backend/app/db/database.py</read_first>
  <action>
Update `backend/alembic/env.py` so it imports settings/Base from canonical paths where safe and imports the model package for metadata registration.

Rules:

- Keep `target_metadata = Base.metadata`.
- Ensure importing `app.models` registers all model modules on `Base.metadata`.
- Do not rewrite existing migration history.
- Preserve current `DATABASE_URL` override behavior.
  </action>
  <acceptance_criteria>

    - `Base.metadata.tables` contains existing table names after import
  </acceptance_criteria>
  <verify>`cd backend && rtk proxy python -c "from app.db.database import Base; import app.models; assert 'users' in Base.metadata.tables; assert 'tasks' in Base.metadata.tables"`</verify>
  <done>Alembic can still discover model metadata after the package split.</done>
</task>
