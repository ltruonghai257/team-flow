# Phase 20 Plan 03: Domain Schema Package Split

<objective>
Convert the large backend schema module into domain schema modules while keeping `app.schemas` as a temporary aggregate compatibility surface.
</objective>

<task id="20-03-01" type="execute">
  <name>Map schemas to domain modules</name>
  <files>backend/app/schemas/</files>
  <read_first>backend/app/schemas.py, backend/app/models/, backend/app/routers</read_first>
  <action>
Create a schema split map matching the model-domain clusters from Plan 02:

- `users.py`: auth/user/invite account schemas where appropriate
- `teams.py`: sub-team and reminder-setting schemas
- `work.py`: projects, milestones, sprints, tasks, statuses, transitions, timeline where tightly coupled
- `notifications.py`: schedule and event notification schemas
- `communication.py`: chat/channel schemas
- `ai.py`: AI conversation/message/project-summary schemas
- `dashboard.py`: dashboard/performance/KPI schemas if they are too large for `work.py`

Use shared/common helpers only when there is actual duplication. Do not create abstract schema frameworks.
  </action>
  <acceptance_criteria>
    - Schema split follows domain/model areas
    - Shared/common module is only introduced if needed
    - No API response field names are changed
  </acceptance_criteria>
  <verify>`rtk proxy rg "^class " backend/app/schemas.py` before moving, then compare representative class imports after the package split</verify>
  <done>Schema split map is clear before moving definitions.</done>
</task>

<task id="20-03-02" type="execute">
  <name>Replace `schemas.py` with `schemas/` package and aggregate exports</name>
  <files>backend/app/schemas.py, backend/app/schemas/__init__.py, backend/app/schemas/*.py</files>
  <read_first>backend/app/schemas.py, backend/app/models/</read_first>
  <action>
Move Pydantic schemas from `backend/app/schemas.py` into `backend/app/schemas/` domain modules.

Important filesystem note: the final import surface should be the package `backend/app/schemas/__init__.py`. Avoid keeping a competing `backend/app/schemas.py` implementation file.

Implementation requirements:

- Preserve schema class names, field names, aliases, validation behavior, model config, and response shapes.
- Import enums/models from canonical model modules where safe.
- Add `backend/app/schemas/__init__.py` aggregate exports for every schema previously imported from `app.schemas`.
- Keep `from app.schemas import TaskOut` and similar old imports working.
  </action>
  <acceptance_criteria>
    - `from app.schemas import UserOut, TaskOut, StatusSetOut, NotificationOut` works
    - Canonical domain schema imports work
    - No Pydantic field names or validation semantics change
    - Existing routers can still import schemas
  </acceptance_criteria>
  <verify>`cd backend && rtk proxy python -c "from app.schemas import UserOut, TaskOut, StatusSetOut, NotificationOut; from app.schemas.work import TaskOut as CanonicalTaskOut; assert TaskOut is CanonicalTaskOut"`</verify>
  <done>`app.schemas` is a package facade backed by domain schema modules.</done>
</task>
