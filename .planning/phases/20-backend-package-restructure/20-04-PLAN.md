# Phase 20 Plan 04: Router Imports and Selected Helper Extraction

<objective>
Move backend routers and tests toward canonical imports, preserve flat domain router registration, and extract only low-risk helpers from the largest routers where that improves maintainability without changing API behavior.
</objective>

<task id="20-04-01" type="execute">
  <name>Update router imports toward canonical modules</name>
  <files>backend/app/routers/*.py</files>
  <read_first>backend/app/routers, backend/app/models/, backend/app/schemas/, backend/app/utils/, backend/app/core/, backend/app/db/, backend/app/socket/</read_first>
  <action>
Update router imports to use canonical modules where the mapping is clear:

- config/limiter/database/auth/AI/email imports should use canonical runtime groups from Plan 01
- model imports should use canonical domain modules where that does not make a router harder to read
- schema imports should use canonical domain modules where that does not create noisy churn
- WebSocket router should import the manager from `app.socket.manager`

Keep aggregate `app.models` and `app.schemas` imports in routers only when canonical domain imports would be ambiguous or too noisy for this phase. Do not change route prefixes, tags, endpoint paths, dependencies, status codes, response models, or request/response behavior.
  </action>
  <acceptance_criteria>
    - Router modules compile
    - `app.api.main` router registration still imports every router
    - `/api/*` paths and `/ws/chat` remain registered
    - No endpoint behavior changes are intentional
  </acceptance_criteria>
  <verify>`cd backend && rtk proxy python -c "from app.api.main import app; paths = {getattr(r, 'path', '') for r in app.routes}; assert '/health' in paths; assert '/ws/chat' in paths; assert any(p.startswith('/api/tasks') for p in paths)"`</verify>
  <done>Routers use canonical imports where safe and still register all current routes.</done>
</task>

<task id="20-04-02" type="execute">
  <name>Extract low-risk helpers from large routers only when useful</name>
  <files>backend/app/routers/tasks.py, backend/app/routers/statuses.py, backend/app/routers/performance.py, backend/app/routers/websocket.py, backend/app/services/</files>
  <read_first>backend/app/routers/tasks.py, backend/app/routers/statuses.py, backend/app/routers/performance.py, backend/app/routers/websocket.py, backend/app/services/</read_first>
  <action>
Inspect the largest routers:

- `performance.py`
- `statuses.py`
- `tasks.py`
- `websocket.py`

Extract helper-only code into `backend/app/services/` or another already-established canonical group only when all of these are true:

- the helper is not itself an endpoint
- extraction reduces import clutter or file size meaningfully
- behavior can be covered by existing focused tests
- the move does not require route registration changes

Do not split routers into product-area subrouters in Phase 20 unless the Phase 19 backend map explicitly calls for it. Endpoint modules should remain flat by domain by default.
  </action>
  <acceptance_criteria>
    - Any helper extraction is behavior-preserving
    - Endpoint decorators remain in the same router file unless a strongly justified split is documented
    - If no extraction is low-risk, the task records that decision in the migration guide later instead of forcing a split
    - Existing tests still pass for touched routers
  </acceptance_criteria>
  <verify>`rtk pytest backend/tests/test_tasks.py backend/tests/test_status_sets.py backend/tests/test_performance.py -q` for touched areas, plus `rtk proxy python -m compileall backend/app`</verify>
  <done>Large routers are either lightly cleaned by helper extraction or intentionally left flat with rationale.</done>
</task>

<task id="20-04-03" type="execute">

  <files>backend/tests/</files>
  <read_first>backend/tests, backend/app/models/, backend/app/schemas/, backend/app/api/main.py</read_first>
  <action>
Update backend tests toward canonical import paths where clear, while keeping focused compatibility coverage in `test_package_structure.py`.

Rules:

- Tests that exercise app startup should import `app.api.main` unless they intentionally verify `app.main`.
- Tests that need specific models/schemas may use canonical domain modules.
- Keep explicit compatibility assertions for `app.main`, `app.models`, and `app.schemas`.
- Do not rewrite unrelated test logic or fixtures.
  </action>
  <acceptance_criteria>
    - Tests compile
    - Compatibility coverage remains for high-risk old paths
    - Most new structure checks use canonical imports
    - No unrelated test behavior is changed
  </acceptance_criteria>
  <verify>`rtk pytest backend/tests/test_package_structure.py -q` and `rtk pytest backend/tests -q` if dependencies are available</verify>
  <done>Tests validate the new package shape while retaining critical old-path compatibility checks.</done>
</task>
