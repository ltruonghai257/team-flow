<task id="17-01-04" type="execute">
  <name>Add backend settings and proposal endpoints</name>
  <read_first>backend/app/routers/sub_teams.py, backend/app/auth.py, backend/app/schemas.py</read_first>
  <action>
Extend `backend/app/routers/sub_teams.py` or create a focused adjacent router if local router registration patterns require it.

Add endpoints for the active sub-team context:

- `GET /api/sub-teams/reminder-settings/current`
- `PATCH /api/sub-teams/reminder-settings/current` for admins only
- `POST /api/sub-teams/reminder-settings/proposals` for supervisors
- `GET /api/sub-teams/reminder-settings/proposals` for admins to list pending proposals
- `POST /api/sub-teams/reminder-settings/proposals/{proposal_id}/review` for admins to approve or reject

Rules:

- Use `get_sub_team` so admins operate on the global switcher's active sub-team and supervisors operate on their own sub-team.
- Members may read current settings but cannot update or propose.
- Admin direct updates immediately change settings.
- Supervisor proposals remain pending until approval.
- On proposal creation, create an `EventNotification` for admin users with `event_type = reminder_settings_proposal`, `event_ref_id = proposal.id`, `remind_at = now`, `status = pending`, and title copy pointing admins back to `/team`.
- On approval, apply only the proposed non-null values to settings and mark the proposal approved.
  </action>

    - `backend/app/routers/sub_teams.py` or a registered router contains `reminder-settings/current`
    - Admin PATCH requires `UserRole.admin`
    - Supervisor proposal path does not mutate `SubTeamReminderSettings`
    - Proposal creation creates `EventNotification`
    - Approval path updates `SubTeamReminderSettings`
  </acceptance_criteria>
