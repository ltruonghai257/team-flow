> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|

## Spacing Scale

Declared values (multiples of 4):

| Token | Tailwind Class | Value | Usage |
|-------|---------------|-------|-------|
| xs | `gap-1` / `p-1` | 4px | Calendar cell inner spacing, chip icon gaps |
| sm | `gap-2` / `p-2` | 8px | Inline metadata spacing, chip padding, segmented controls |
| md | `gap-4` / `p-4` | 16px | Default form spacing, card rows, page legend rows |
| lg | `gap-6` / `p-6` | 24px | Main page blocks, tab content spacing |
| xl | `gap-8` | 32px | Desktop page layout breathing room |
| 2xl | `py-12` | 48px | Empty-state vertical rhythm |
| 3xl | `py-20` | 80px | Full-page loading state centering |

Exceptions:
- Shared `.card` stays `p-5` (20px) because the existing scheduler already uses it.
- Session tiles inside agenda lists use `p-3` to keep upcoming content dense and scannable.
- Icon-only controls must stay at least 36x36px hit area.

## Color

| Role | Tailwind Token | Hex | Usage |
|------|---------------|-----|-------|
| Dominant (60%) | `bg-gray-950` | #030712 | App background, page canvas |
| Secondary (30%) | `bg-gray-900`, `bg-gray-800`, `border-gray-800` | #111827, #1f2937 | Cards, modals, tab panels, agenda tiles |
| Accent (10%) | `primary-500`, `primary-600` | #6366f1, #4f46e5 | Active tab, primary CTA, selected reminder chips, focus ring |
| Destructive | `red-600` | #dc2626 | Delete/cancel session actions only |

Accent reserved for:
- Active tab pill or underline
- `Create Session` / `Save Changes` primary CTA
- Selected reminder chips
- Active calendar day ring
- Knowledge session event color in calendar dots and left borders

Secondary semantic colors:
- Org-wide scope badge: `bg-blue-500/15 text-blue-300 border-blue-500/30`
- Sub-team scope badge: `bg-emerald-500/15 text-emerald-300 border-emerald-500/30`
- Session type badges:
  `presentation` = indigo tint,
  `demo` = cyan tint,
  `workshop` = amber tint,
  `qa` = rose tint

Do not recolor personal schedule events. Knowledge Sessions use a distinct indigo-blue accent family while task due dates remain amber.

## Interaction Contract

### 1. Page Structure

The page stays on `/schedule`. No new route or new sidebar item is added.

Desktop layout remains a 2-column calendar + upcoming rail composition. The Knowledge Sessions experience is introduced as an upper-level segmented tab inside the page header area:

```text
/schedule
├── Page header
│   ├── Left: "Scheduler" + subtitle
│   └── Right: CTA area
│       ├── For members: existing "New Event" only
│       └── For supervisors/admins: active-tab-aware CTA
├── Primary tab bar
│   ├── My Schedule
│   └── Knowledge Sessions
└── Tab panel
    ├── My Schedule: current experience preserved
    └── Knowledge Sessions
        ├── Toolbar row
        │   ├── view toggle: Agenda / Calendar
        │   ├── optional sub-team context hint
        │   └── create CTA for supervisors/admins
        ├── main board
        │   ├── agenda list by default
        │   └── calendar-grid alternative view
        └── side rail
            └── upcoming sessions list or selected-day session list
```

The tab switch is not decorative. It partitions state. Personal schedule modal state and knowledge session modal state must be completely separate.

### 2. Tab Bar

Use a 2-option segmented control directly under the page intro.

- Wrapper: `inline-flex rounded-lg border border-gray-800 bg-gray-900 p-1`
- Tabs: equal-height buttons, `text-sm font-medium px-4 py-2`
- Active tab:
  `bg-primary-600 text-white shadow-sm`
- Inactive tab:
  `text-gray-400 hover:text-gray-200`

Tab labels:
- `My Schedule`
- `Knowledge Sessions`

When switching tabs:
- Close whichever modal is open.
- Reset edit target for the inactive domain.
- Preserve month navigation state.
- Preserve fetched data caches if already loaded.

### 3. Knowledge Sessions Default View

The Knowledge Sessions tab opens in `agenda` view by default per D-02.

The toolbar directly above the content contains:
- left: `Knowledge Sessions` panel title and a compact scope explanation
- center/right: `Agenda` / `Calendar` view toggle
- far right: `Create Session` button for supervisor/admin only

View toggle style:
- Same segmented pattern as the tab bar but smaller.
- `Agenda` uses `List` icon + text.
- `Calendar` uses `Calendar` icon + text.

Copy:
- Section title: `Knowledge Sessions`
- Helper copy:
  `Org sessions and your team's scheduled knowledge sharing.`

### 4. Agenda View

Agenda is the primary browsing experience and should feel like an operations list, not a marketing card deck.

Layout:
- vertical stack of session tiles in chronological order
- grouped by month/day separators only if there are enough sessions to benefit from it
- each tile uses a compact framed row style inside `bg-gray-800 border border-gray-700 rounded-lg p-3`

Session tile content order:
1. Topic
2. Start time + duration
3. Presenter name
4. Scope badge and session type badge
5. Tags row
6. Description preview (2 lines max if long)
7. References teaser line or count
8. Action row for supervisors/admins on sessions they can manage

Tile behavior:
- Entire tile is clickable to open detail modal.
- Hover adds subtle border and background shift only.
- No oversized illustrations or marketing empty space.

### 5. Calendar View

Calendar view is a secondary switch, not a separate route.

It reuses the existing month grid style from `/schedule`, but Knowledge Sessions events display differently from personal schedules:
- left border or dot uses indigo-blue KS color
- task due dates stay amber
- personal schedule events stay their current colors

In Knowledge Sessions calendar mode:
- clicking a day does not open a personal event form
- clicking a day filters the side rail to that date's sessions and offers `Create Session` seeded to that date for authorized users
- if a date has no sessions, the side rail shows an empty day-state with creation CTA for authorized users

Event density rules:
- show max 2 session labels inside each day cell
- overflow becomes `+N more`
- title truncates after one line

### 6. Side Rail / Detail Rail

The right-hand card becomes active-tab-aware.

For `My Schedule`:
- keep current `Upcoming` experience unchanged

For `Knowledge Sessions`:
- title becomes `Upcoming Sessions` in agenda mode
- title becomes `Selected Day` in calendar mode when a day is chosen
- items use the same SessionCard compact format as agenda tiles but without full description unless selected

If no sessions exist:
- heading: `No knowledge sessions yet`
- body: `Supervisors and admins can schedule the first session for this team.`
- for members, remove the action sentence and end with `Sessions will appear here when scheduled.`

### 7. Session Form Modal

Session creation/editing uses a dedicated modal. It must not reuse the existing personal `form` object or `showModal`.

Suggested state names:
- `showKnowledgeModal`
- `editingKnowledgeSession`
- `knowledgeForm`
- `knowledgeReminderOffsets`
- `knowledgeTagDraft`

Modal shell matches current scheduler modal:
- `fixed inset-0 bg-black/60 backdrop-blur-sm`
- panel `bg-gray-900 border border-gray-800 rounded-xl w-full max-w-xl`

Form field order:
1. Topic
2. Description
3. References
4. Presenter
5. Session type
6. Duration
7. Date/time
8. Tags
9. Reminder offsets

Field rules:
- Topic is required.
- Presenter defaults to current user.
- References is a textarea, not structured link rows.
- Tags use chip entry, not comma-separated raw input.
- Reminder offsets reuse current chip pattern from personal schedules.

### 8. Tags Input

Tags are entered through a lightweight chip composer:

```text
[existing chips...] [input with placeholder]
```

Behavior:
- Enter or comma commits a chip
- duplicate chips are ignored case-insensitively
- trim whitespace
- max visible chip row wraps naturally
- removing a chip uses a small `X` icon button

Placeholder:
- `Add tags`

Chip style:
- `inline-flex items-center gap-1.5 px-2.5 py-1 rounded-full bg-primary-600/15 border border-primary-500/25 text-primary-200 text-xs`

### 9. Presenter Field

Presenter uses a select control with scope-aware options.

Rules:
- Admin org-wide context can choose any active user.
- Supervisor sees only their sub-team members.
- Presenter defaults to the session creator.
- Empty presenter is not allowed in UI; there is always a current selection.

Helper text:
- For supervisor:
  `Presenter must belong to your sub-team.`
- For admin with no team filter:
  `Org-wide sessions can use any active presenter.`

### 10. References Field

References stays intentionally low-friction:
- multi-line textarea
- accepts notes, pasted URLs, and simple resource lists
- rendered as plain text in modal/detail surfaces for this phase unless safe linkification is deliberately added in execution

Placeholder:
`Paste links, docs, or prep notes for attendees`

No markdown rendering is required for this phase.

### 11. Reminder Chips

Reminder selector matches the personal schedule pattern for consistency.

Available defaults:
- `15 min before`
- `30 min before`
- `1 hour before`
- `1 day before`

Selected chip style:
- `bg-primary-600/20 border-primary-500 text-primary-200`

Unselected chip style:
- `bg-gray-800 border-gray-700 text-gray-400 hover:text-gray-200`

Supporting copy:
`Everyone in scope receives the selected reminders.`

### 12. Permissions and Actions

Member behavior:
- can browse sessions in scope
- cannot see `Create Session`
- cannot see edit/delete controls

Supervisor behavior:
- sees `Create Session`
- can edit/delete sessions they are allowed to manage
- presenter options limited to their sub-team

Admin behavior:
- sees `Create Session`
- can manage all sessions
- global view may reflect selected sub-team context from the existing admin team switcher

Delete interaction:
- modal-level or inline confirmation is acceptable, but must use explicit product copy
- avoid browser `confirm()`

Recommended confirmation copy:
- Title: `Delete session`
- Body: `This removes the session and its pending reminders for people in scope.`
- Confirm button: `Delete Session`

### 13. Loading, Empty, and Error States

Initial tab load:
- use the existing centered spinner pattern

Agenda empty state:
- heading: `No knowledge sessions yet`
- body for authorized users:
  `Schedule a session to share upcoming demos, workshops, or Q&A time.`
- body for members:
  `Sessions for your scope will appear here when they're scheduled.`

Filtered day empty state in calendar mode:
- heading: `Nothing scheduled for this day`
- body:
  `Choose another date or create a new session here.`

Error toast examples:
- `Failed to load knowledge sessions.`
- `Failed to save session. Try again.`
- `Failed to delete session. Try again.`
- `Reminders were not updated.`

### 14. Mobile Contract

On mobile:
- tab bar wraps only if needed, but should prefer horizontal compact buttons
- agenda list becomes the default usable mode
- calendar view remains available but side rail content stacks below the grid
- create CTA stays visible at the top of the tab panel, not buried under the calendar
- form modal becomes near-full-screen height with internal scrolling

No card may rely on hover-only affordances for essential actions on mobile.

## Visual Hierarchy

### Page Priority Order
1. Scheduler page header
2. Tab bar
3. Active tab toolbar
4. Active content surface
5. Side rail / detail rail

### Knowledge Tab Priority Order
1. Create/browse mode selection
2. Session topic and time
3. Presenter and scope
4. Description and references
5. Admin/supervisor actions

### Density Guidance

This feature should feel like part of a work tool:
- dense but readable
- restrained card radius
- quiet background
- strong scanning over decorative flourish

Avoid:
- oversized hero-like headers
- large empty cards
- floating nested cards inside cards
- decorative gradients or illustrations
